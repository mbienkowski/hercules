# Requirements ↔ Code Index for Hercules — Research & Plan

## Context

Hercules ends its workflow with only **business requirements + code** surviving (specs are
retired to avoid drift). Two problems follow:

1. **Traceability is expensive to reconstruct.** Whenever a user asks about a bug/feature/flow
   change, Claude must re-scan the whole repo and all requirements to connect them. Git-history
   archaeology helps only when docs and code share a repo — it breaks for a centralized docs
   repo + distributed microservice repos.
2. **Brownfield repos have no requirements at all.** 99% of repos start as code-only; they need
   IDE-style project indexing so the agent can navigate cheaply, and a way to grow
   requirement links over time.

The user's key constraint: the index must live **in the project repo as normal git files** so
that 100 people don't each pay thousands of dollars re-indexing the same repo — index once,
share via git. Which immediately raises merge-conflict, branching, and hook-automation concerns.

## Business requirements (distilled from the discussion)

### A. Bidirectional traceability (problem #1)
- **BR-A1** Given a requirement (ID or topic), Hercules must resolve the implementing code
  locations (repo, paths, symbols) quickly and cheaply — without a full-repo scan.
- **BR-A2** Given a code location (file/symbol), Hercules must resolve the governing business
  requirement(s) equally fast.
- **BR-A3** Works across repo boundaries: requirements may live in a centralized docs repo
  while code lives in N microservice repos.
- **BR-A4** When a change (bug fix, feature change, flow change) is delivered, both the code and
  the linked requirements must be discoverable so both get updated together; the index itself
  is updated as part of delivery.
- **BR-A5** The link store is consumed by Claude/Hercules only (not human-facing docs); it may
  be machine-oriented, shorthand, token-optimized.

### B. Brownfield indexing (problem #2)
- **BR-B1** Existing code-only repos can be indexed to produce the same navigational layer
  (files, symbols, modules, entry points) that greenfield Hercules projects get incrementally.
- **BR-B2** Two indexing depths:
  - **Full**: recommended for small repos — one-shot, in-depth.
  - **Lightweight/adaptive**: for large monoliths — high-level everywhere, deep only where it
    matters (hot paths, recently-changed areas, user-selected subsystems).
- **BR-B3** Where no requirements exist, code can be distilled into "in-file knowledge"
  (behavioral summaries) that later seeds real requirements.

### C. Cost & mechanics
- **BR-C1** The structural layer must be generated by **deterministic static analysis** (no LLM
  tokens); LLM effort is reserved for semantic enrichment only.
- **BR-C2** Query must be tool-driven (grep/CLI/lookup), reading small slices — never "load the
  whole index into context".
- **BR-C3** Index format is token-cheap (e.g. TOON/tabular/shorthand rather than verbose JSON).

### D. Git-native sharing
- **BR-D1** The index is committed to the project repo as plain files; a fresh clone + Hercules
  has the "library of context" instantly, zero re-indexing cost.
- **BR-D2** Merge-friendly: no single-file bottleneck; branch work auto-indexes before commit
  (hook), and merges combine knowledge without manual conflict surgery.
- **BR-D3** Indexing of your own branch changes happens automatically (pre-commit or
  workflow-step hook), so every commit carries current knowledge.
- **BR-D4** No bespoke database engine — compose existing open-source, free tooling.

### Non-goals (proposed)
- Not a human documentation system (requirements stay the human-facing artifact).
- Not a full IDE language server; "good enough to route the agent" beats "perfect semantics".
- Not real-time; index freshness is per-commit granularity.

## Research findings

### Hercules codebase (branch `claude/hercules-workflow-analysis-m4jaav`)

**What already exists — the feature is mostly "persist what's already computed":**

- **The requirement↔code edge is already produced, then thrown away.** Specs carry
  `satisfies: [<file> §Section]` headers; Build's checkpoint (`build.md` step 9) records
  `satisfies` links + `tests_added` + interfaces into `~/.hercules/state/{slug}.json`
  (`build_progress[]`), and the G5 cross-check computes a full bidirectional matrix
  (`✓ requirement → evidence`) — but only prints it. Then the spec is `git rm`'d (step 10).
  Post-delivery the mapping survives **only in machine-local state**, never in the repo.
- **No requirement IDs.** Requirements are addressed by `## Section` heading in
  `docs/{session}/*-business-requirements.md`. A stable ID scheme is a prerequisite for a
  durable index (ripples into `discover.md` template, `design.md` implementability check,
  `satisfies:` grammar, learnings keying).
- **Multi-repo is modeled.** `~/.hercules/config.json` has a per-project `repositories`
  map (service → local path) and `docs_root` may be a **separate docs repo**
  (CLAUDE.md principle 6). `hercules_state.py:resolve_session()` already resolves all roots —
  the natural shared library for any new hook.
- **In-repo committed conventions to build on:** `docs/INDEX.md` (session table),
  `docs/learnings.md` (reconciled, id-keyed, atomic temp+rename store — architecturally the
  sibling of our index), the `satisfies:` header grammar. Machine-local state is *never*
  written into the repo today, so a committed index is a new artifact class but follows the
  learnings pattern.
- **Hooks:** only `PreToolUse` (frozen-tests guard) exists; `PostToolUse`/`Stop` unused;
  Ship does real `git commit` — several clean places to auto-update the index. Today's hooks
  are read-only & fail-open; a writing hook is a new permission posture to document.
- **Token budgets are enforced infrastructure:** `tests/metrics/token_counter.py` (tiktoken),
  `threshold_runner.py` + `tests/testdata/thresholds.json` — adding a token cap on the index
  file is a one-line row. The A2A "minimal slice" delegation packet is the natural consumer:
  the index resolves requirement §↔ files ↔ tests so orchestrators assemble tighter slices.

**Best extension points (lowest disruption first):**
1. Build step 9/10 (checkpoint/retire) — write the committed index entry exactly when the
   spec (the bridge document) is about to be deleted.
2. G5 cross-check — persist the matrix it already computes.
3. Design step 3 codebase scan ("Affected code") — seed for brownfield indexing.
4. New skill modeled on `learnings` (reconciled, id-keyed, ≤900 tokens/≤20 blocks).
5. `hooks.json` PostToolUse/Stop for incremental freshness; Ship staging for atomic commits.

### Tooling landscape (what exists OSS/free — no bespoke DB needed)

**2026 consensus for coding agents: structural/lexical retrieval beats frozen embeddings.**
Claude Code itself uses live glob/grep/read with no persistent index; Aider uses tree-sitter +
PageRank repo-map (no embeddings); Cursor's embeddings are a server-side *cache*, never
committed. Committed vector stores are an anti-pattern: binary, high-churn, unmergeable,
stale on every edit, often non-deterministic builds.

**Deterministic structural indexers (no LLM tokens):**
- **tree-sitter** (MIT) — real AST, broad languages, the de-facto 2026 structural layer.
- **universal-ctags** (GPL-2 standalone binary) — zero-config symbol tags, cheapest fallback.
- **SCIP** (Apache-2.0, Sourcegraph) — precise cross-file refs, per-language indexers, heavier;
  LSIF is dead. Glean/Kythe/stack-graphs: too heavy or too narrow for a plugin hook.

**Requirements traceability prior art:**
- **OpenFastTrace** anchor convention — `[impl->req~id~rev]` tags in code/test comments;
  `type~name~revision` IDs give staleness detection. Adopt the *convention* (the GPL binary
  isn't needed — tags harvest with a deterministic grep pass).
- **Doorstop** — one-file-per-requirement sharding = merge-friendly by construction.

**Git-native merge mechanics:**
- **Sharded one-file-per-entity** (towncrier changelog-fragment pattern) → near-zero conflicts.
- **`merge=union`** in `.gitattributes` for append-only line-per-record aggregates (pair with
  deterministic sort+dedupe on regeneration).
- **"Regenerate on conflict" custom merge driver** as backstop for derived files.
- **git notes rejected** — not fetched/pushed by default; fights the "clone = instant context" goal.
- Binary search indexes (SQLite FTS5, zoekt, tantivy) are **rebuild-locally, gitignore** artifacts;
  commit only derived text.

**Token-cheap serialization:**
- **TOON** (v3.2 working draft) — ~40% fewer tokens than JSON at equal-or-better LLM retrieval
  accuracy on uniform tables (arXiv 2603.03306); plain CSV/TSV is even smaller for flat tables;
  JSON only for irregular structure. Index rows (`req_id, file, symbol, kind`) are exactly the
  uniform-tabular sweet spot.

**Optional semantic layer:** commit only text summaries; rebuild embeddings locally cached by
content hash (sqlite-vec / usearch, gitignored) — everyone gets semantics without shipping vectors.

## Proposed approach

**Thesis: don't build an indexer first — stop discarding an edge Hercules already verifies.**
Design's coverage gate and Build's checkpoint/G5 cross-check already compute the verified
requirement↔code↔test matrix; today it dies with the retired spec. Persist it as sharded,
committed, token-cheap text files (the `learnings.md` contract, sharded per entity for merge
safety), then extend the same file layer to brownfield repos via a deterministic scanner.

### 1. Index anatomy (two locations, sharded text, no binaries)

```
# docs repo ({docs_root})
docs/trace/
  README.md                      # 10-line human explainer
  manifest.json                  # schema_version, project slug, indexed repos
  req/user-auth.token-storage.md # ONE FILE PER REQUIREMENT (forward shard)

# each code repo root
.trace/
  manifest.json                  # scan mode (full|light), scanned-at commit
  map/src__auth.tsv              # ONE FILE PER SOURCE DIR (reverse shard)
  notes/src__auth.md             # LLM "in-file knowledge" (brownfield seed)
```

**Forward shard** (`docs/trace/req/<id>.md`) — markdown, write-once 2-line front matter
(`id`, `source`) + append-only pipe table (~110 tokens each):

```markdown
---
id: 2026-06-22-user-auth.token-storage
source: 2026-06-22-user-auth/2026-06-22-user-auth-business-requirements.md#token-storage
---
| repo | kind | path | ref | rev |
|---|---|---|---|---|
| svc-auth | impl | src/auth/token_repo.py | TokenRepository.save | 1 |
| svc-auth | test | tests/auth/test_login.py | test_token_persists | 1 |
```

The `rev` column records which requirement revision the evidence was verified against —
without it, nothing *committed* would carry the rev after the spec (the only other carrier)
is deleted, and staleness detection would be impossible from a fresh clone. It's written once
per row (append-only, no churn).

**Reverse shard** (`.trace/map/*.tsv`) — TSV rows `path  symbol  kind  reqs`.

**Format verdict on TOON:** measured against the actual data shapes, **TSV beats TOON** on
flat tables (TOON's ~40% win is vs JSON; CSV/TSV is smaller still), TOON v3.2 is a working
draft with no stdlib parser (would break the plugin's stdlib-only hygiene tests), and the
forward shards benefit from being human-reviewable markdown in PRs. So: markdown tables for
the human-reviewed forward shards, TSV for machine-only reverse shards, JSON only for the tiny
manifests. Revisit TOON if shards ever grow nesting.

### 2. Stable requirement IDs (prerequisite)

- **ID = `{full-session-slug}.{section-slug}`** including the date
  (e.g. `2026-06-22-user-auth.token-storage`) — no central counter file, so no allocation
  conflicts, and genuinely unique by construction: session dirs are only unique *because of*
  the date prefix, so a date-less ID would collide the first time the same area is revisited
  (`2026-09-10-user-auth` would re-mint `user-auth.token-storage`). Matches how everything
  else in Hercules is keyed. Discover's shard bootstrap additionally collision-checks
  `docs/trace/req/` and, on a topical match, offers "amend the existing requirement (bump rev)
  instead of minting a new ID" (see the amend branch below).
- **Anchor = invisible HTML comment** under each requirement section:
  `<!-- req: user-auth.token-storage rev:1 -->` — doesn't render for stakeholders (preserves
  the "business language only" constraint), greppable, survives heading rewording.
- **OFT-style `rev:N`** bumped on every section edit; specs record `satisfies: [<id>@<rev>]`;
  `@rev` behind the anchor = suspect/stale link, surfaced not auto-fixed.
- **Legacy docs resolve without edits** — shard `source:` uses deterministic heading slugs;
  an optional migration pass inserts anchors into existing requirement files.

### 3. Write paths

One deterministic **stdlib-only script** `plugin/skills/traceability/scripts/trace.py`
(subcommands `scan`, `add-evidence`, `query`, `check [--fix]`; atomic temp+rename, idempotent
dedupe+sort) does all shard I/O. Commands just invoke it.

**Greenfield (workflow insertions):**
- `discover.md` step 7: write ID anchors + empty shards; bootstrap `docs/trace/` + a canonical
  `.gitattributes` union line on first use. **New amend branch at step 1:** grep the incoming
  idea against `docs/trace/req/` IDs; on a match, offer "this matches requirement `<id>` —
  amend that section (bump `rev`, refresh shard) instead of opening a new session?" The
  rev-bump rule lives at *every* site that edits requirement files (this amend branch + Build's
  G5 drift-amend disposition), not only in the skill.
- `design.md` step 7/9: `satisfies:` validates against IDs/shards; template uses `[<id>@<rev>]`.
- `build.md` **step 7 (traceability gate G4)** — not step 9: `trace.py add-evidence` persists
  the evidence *at the moment it is proven*, before step 8's Advance prompt. Critical because
  the **ship-each cadence fires at step 8** — writing at step 9 would mean every spec-scoped
  commit lands *without* its shard. The shard is the committed twin of the machine-local
  checkpoint (step 9 still writes that).
- `build.md` step 3 (write failing tests): test-docstring `satisfies:` anchors are authored
  **here, pre-freeze, via the `write-test-scenarios` skill** (added to the touch list).
  `trace.py` never writes into test files — `scan` only *extracts* anchors — so the frozen-test
  guard (G1) is never bypassed and the pre-advance diff backstop never wedges.
- `build.md` G5 cross-check: `trace.py check --fix` — the verified matrix is committed, not
  just printed. **G4 stays `prompt-only`** (the class taxonomy is test-pinned as an honesty
  label); the trace check is documented as a *necessary-condition executable sub-check* (row
  exists + named test exists — it cannot verify passing or semantic honesty).
- `ship.md` staged set: `docs/trace/**` + `.trace/**` alongside the existing INDEX.md clause;
  `trace.py check --changed` regenerates stale map shards deterministically at ship time.
  Two explicit amendments the current ship.md lacks: (a) the **separate docs repo (`docs_root`)
  becomes a named commit target** — it is not in the `repositories` map, and today ship reaches
  it only implicitly via the INDEX.md clause; (b) the **spec-scoped staged set** (and the G6
  registry row) explicitly includes "this spec's `docs/trace/req/*.md`".
- **Consistency model, not atomicity:** cross-repo commits are sequential (ship.md: push in
  delivery order, stop on first failure) — a single atomic code+shard commit is impossible when
  docs live in a separate repo. Order: code repos first, docs repo last; `trace.py check`
  treats both shard-ahead-of-code and code-ahead-of-shard as **WARN, never error**.

**Brownfield — new `/hercules:trace` command** (plan-mode wizard like all commands; named
`trace` to match the dirs and avoid colliding with the existing `docs/INDEX.md` artifact):
1. Resolve project + `repositories` map; pick repos.
2. Mode by size (`git ls-files` count): **full** ≤ ~2k source files, else **light**
   (directory-level rows; deepens adaptively when later sessions touch an area). User overrides.
3. **Deterministic pass (zero LLM tokens):** stdlib regex symbol extraction per language
   family; auto-upgrade to user-installed ctags/tree-sitter if detected (never bundled/required).
4. **LLM enrichment pass (consented):** 2–4 sentence business summaries per area into
   `notes/*.md`, flagging "no requirement governs X" — seeds for future Discover sessions.
5. Optional legacy-requirements annotation pass; hand off to `/hercules:ship`.

**Freshness — v1 ships NO writing hook** (today's hooks are read-only/fail-open; that posture
is a trust asset). Freshness = workflow writes + ship-time deterministic regeneration.
**The read-only Stop-hook advisory moves INTO v1** ("you edited files traced to req X — update
the requirement + bump rev?"): skills are model-invoked by description, so a plain "fix this
bug" conversation would never load the traceability skill — the advisory is the *only*
mechanism that reaches ad-hoc conversations, and it preserves the read-only hook posture.
Without it, the v1 bug-fix contract would be a rule written in a skill that never fires on the
path where bugs actually get fixed. Out-of-band commits: documented opt-in pre-commit snippet;
Hercules never installs git hooks itself.
**No per-row content hashes** (they'd churn every line and sabotage union merges); staleness =
git-changed-files ∩ map rows, plus rev counters for requirements.

### 4. Read paths

New **`traceability` skill** (≤900 tokens/≤20 blocks, learnings-style contract):
- req→code: read exactly `docs/trace/req/{id}.md`, open only the listed paths. Never read the
  whole tree.
- code→req: one `rg` over `docs/trace/req/` or one `.trace/map/{dir}.tsv` read → IDs → shards
  → governing sections. No store = clean no-op.
- **Bug-fix contract stated once in the skill:** behaviour change → update requirement + bump
  rev + refresh shard; pure defect → add regression test row. (This is BR-A4.)
- Delegation packet `Context:` gains "+ trace shard rows for in-scope req IDs" (~200 tokens
  instead of a repo scan); `design.md` step 3 scan starts from `.trace/map`+`notes`;
  `discover.md` step 0 surfaces matching notes.

### 5. Merge story

- `docs/trace/req/*.md` + `.trace/map/*.tsv`: **`merge=union`** in `.gitattributes`.
  Verified character-level: two branches appending rows to a base-present shard merge cleanly
  (header/front matter are common context; union concatenates only the conflicting end-of-file
  hunk). **Sharding-per-entity is the primary defense; union is the assist** — platform-side
  merges (GitHub/GitLab web UI) don't run local merge drivers, so occasional conflicts remain
  possible there.
- Three failure modes explicitly handled:
  1. **add/add with no merge base** (two branches independently *create* the same shard) —
     union degenerates to whole-file concatenation → doubled front matter/header. Therefore
     `check --fix` canonicalizes the **whole file** (collapses duplicate front-matter/header
     blocks), not just rows, and a post-union add/add fixture goes into `test_trace_script.py`.
  2. **Resurrected rows** (a row modified on one branch, e.g. symbol renamed, survives in both
     old and new form after union): the skill's read path is duplicate-tolerant
     (last-row-wins per `(repo, path, ref)`); `check --fix` reconciles on the next run. G5/ship
     runs `check --fix` after merges touch trace files.
  3. **`check --fix` refuses to run while `git ls-files -u` shows unmerged trace paths** —
     never ingests conflict markers as rows.
- Front matter is *write-once by tools in normal operation*; `check --fix` may repair a
  dangling `source:` when it can re-resolve the anchor comment (the anchor is the identity,
  the path a hint), and reports unresolvable orphan shards as WARN — never deletes them.
- `notes/*.md`, manifests: default merge (conflicts meaningful/rare; manifests regenerable).
- Backstop: optional documented custom merge driver (`trace.py merge-driver`) — regenerate
  derived shards on conflict; never required. v2 must repeat the `.gitattributes` bootstrap +
  consent **per code repo** (for `.trace/map/*.tsv`), emitting a byte-identical canonical line
  so parallel bootstraps add/add-merge cleanly.

### 6. In-code anchors: hybrid — tests yes, impl no

Test docstrings carry `satisfies: <req-id>` (adopting OpenFastTrace's *convention*, not its
GPL binary): the req→test edge is the load-bearing one (G4), tests are Hercules-governed
(frozen list), anchors survive renames, and `scan` re-extracts them → self-healing index.
Production code stays clean; impl maps by path/symbol rows (cheap to regenerate).

### 7. Budgets & tests (reuse existing infra)

- `tests/testdata/thresholds.json`: new `token_count` row for `plugin/commands/trace.md`
  (≤1500); small bumps for touched commands (build/design/discover/ship). For
  workflow-protocol.md (at 999/1000, past its 950 warn line): **raise the ceiling to 1100
  (warn 1050)** — in line with sibling budgets (a2a core 1050, debate 1100) — plus trim the
  preamble/role-expectations prose if the traceability wording still binds. (This replaces the
  earlier "offsetting trims only" note — see the protocol-layout advisory below.)
- `tests/skills/test_skills.py`: add `traceability` to the skill lists (contract auto-applies).
- `tests/commands/test_commands.py`: `/hercules:trace` structure assertions + trace assertions
  in build/design/ship tests.
- New `tests/scripts/test_trace_script.py`: idempotence (double-run = identical bytes),
  post-union-merge dedupe, per-language scan fixtures, staleness/suspect-rev detection,
  stdlib-only hygiene (mirrors `test_hook_hygiene.py`).

### 8. Phasing

- **v1 — greenfield loop:** ID anchors + rev; forward shards (with rev column); `traceability`
  skill + `trace.py` (`add-evidence`/`query`/`check`); the command/protocol insertions incl.
  the discover **amend branch** and the ship docs-repo/spec-scoped clauses; test-anchor
  convention via `write-test-scenarios`; `.gitattributes`; **read-only Stop-hook advisory**;
  thresholds + tests. *(No reverse shards yet — code→req resolves by grepping the small
  `docs/trace/req/` dir, which already works cross-repo.)*
- **v2 — brownfield:** `/hercules:trace` command (renamed from `/hercules:index` — "index"
  collides with the existing, user-visible `docs/INDEX.md` artifact while the dirs are named
  `trace`); `trace.py scan`; `.trace/map` + `.trace/notes`; full/light modes (manifest records
  scan scope/mode **per directory** so never-deepened areas aren't flagged stale and two clones
  can explain divergent coverage); legacy annotation migration (runs only when no session is in
  `current_phase: "build"` — no frozen list in play); per-repo consent with **remembered
  refusals** (a `repositories`-adjacent flag, so ship doesn't re-dirty a declined repo forever).
- **Deferred:** embeddings (only ever as a local gitignored cache keyed by content hash),
  TOON, default custom merge driver, call-graph/PageRank ranking, any mutating hook.

## Real-case scenarios (how a user actually interacts)

All four assume the v1+v2 feature set. "Excluded" = deliberately out of scope for that flow.
Grounding fact: Hercules never skips phases (tier only scales advisor count), so "lightweight"
flows happen either *inside* Discover (the amend branch) or *outside* the pipeline entirely
(ad-hoc chat + skill + Stop-hook advisory).

### S1 — Bug fix, requirements exist

> **User:** "Users report login sessions die after 5 minutes — should be 30. Fix it."

1. Claude greps `.trace/map/src__auth.tsv` (or `docs/trace/req/`) for the touched area → hits
   `2026-06-22-user-auth.session-timeout` → reads that one shard (~110 tokens) → opens exactly
   the governing requirement section, `src/auth/session.py`, and `tests/auth/test_login.py`.
   **No repo scan, no requirements-folder scan.**
2. Fork, stated once in the traceability skill — Claude checks what the requirement says:
   - **Requirement says 30 min → pure defect.** Fix the code, add a regression test carrying
     `satisfies: 2026-06-22-user-auth.session-timeout` in its docstring, append a `test` row
     to the shard. Requirement text and `rev` untouched.
   - **Requirement says 5 min → behaviour change, not a bug.** Claude says so and routes to
     the S3 flow (the requirement must change first).
3. On Stop, the read-only advisory hook double-checks: "you edited files traced to
   `…session-timeout` — requirement updated or confirmed unchanged?" — catching the case
   where the user's phrasing never triggered the skill.

**Expected at the end:** code fix + anchored regression test + one new shard row; requirement
untouched (pure-defect branch); everything ready for a normal commit.
**Excluded:** no new session dir, no INDEX.md row, no spec, no requirement edit, no tier
scoring, no re-index beyond the touched rows, no PR unless asked.

### S2 — Bug fix, no requirements (brownfield)

> **User:** "The CSV export crashes on files with a BOM. Fix it."

1. If the repo was indexed (`/hercules:trace` ran): the map row for `src/export/` shows
   `reqs: -`; `.trace/notes/src__export.md` gives the behavioural summary → Claude navigates
   straight to the right file. If never indexed: today's behaviour (live grep) — the skill
   no-ops cleanly and may suggest running `/hercules:trace`.
2. Fix + regression test — **no anchor** (there is no requirement ID to cite).
3. If the fix revealed durable behaviour knowledge ("export must tolerate BOM"), Claude
   updates the note section and flags: "no requirement governs CSV export — candidate for a
   Discover session." Touched map rows refresh at commit time (ship-time `check --changed`
   or the opt-in pre-commit snippet).

**Expected at the end:** code fix + regression test + refreshed note/map rows + an explicit
"unmanaged area" flag the team can act on later.
**Excluded:** no requirement is authored automatically (requirements are stakeholder
artifacts — Discover owns them), no shard created (no ID exists), no retroactive documentation
of the module, no tier scoring.

### S3 — Change an existing feature, requirements exist

> **User:** "/hercules:discover — checkout should now support partial refunds."

1. **Discover (amend branch):** step 1 greps the idea against `docs/trace/req/` → matches
   `2026-03-02-payments.refunds`. Hercules offers: *"This matches an existing requirement —
   amend that section (bump rev 1→2), or open a new session for net-new scope?"* Typical
   answer: amend the changed section **and** put net-new scope in the new dated session doc.
   INDEX.md gets the new session row; the amended section's existing shard rows now show
   `rev 1 < anchor rev 2` → flagged **suspect** ("this code predates the requirement change"),
   which is exactly Design's work list.
2. **Design:** specs cite `satisfies: [2026-03-02-payments.refunds@2]`, validated against the
   anchor; the coverage gate ensures every amended/new requirement section is owned by a spec.
3. **Build:** failing tests written with anchors (step 3, pre-freeze) → implement → at the G4
   gate (step 7) `trace.py add-evidence` appends rev-2 rows → ship-each commits carry code +
   shard together → spec retired (step 10); the mapping survives the deletion.
4. **Ship (close-out):** stages residue; the docs repo is committed as its own target;
   `check` confirms no remaining suspect rows for the amended section.

**Expected at the end:** amended requirement (rev bumped) + new session doc + INDEX.md row +
shard rows at rev 2 alongside historical rev-1 rows + anchored tests + specs deleted + code
shipped.
**Excluded:** rev-1 rows are not deleted (append-only history; `check --fix` removes only
demonstrably dead rows), other services' notes untouched, cross-repo commits are sequential
(WARN-level skew accepted, never silent), stale rows in *unrelated* requirements not chased.

### S4 — Change an existing feature, no requirements (brownfield)

> **User:** "/hercules:discover — rework CSV export to stream instead of loading into memory."

1. **Discover:** no `docs/trace/req/` match, but `.trace/notes/src__export.md` (from a prior
   `/hercules:trace` run, or from S2's flag) is surfaced at step 0 as seed context — the
   note's behavioural summary becomes the draft's starting point instead of a blank page.
   Discover writes the **first** requirements doc for this area (new dated session dir, ID
   anchors, empty shard) — the area transitions from unmanaged to traced.
2. **Design/Build/Ship:** exactly the S3 flow from step 2 on; the deterministic map rows for
   `src/export/` gain `reqs: 2026-07-15-csv-streaming.*` entries at ship time.
3. Alternative: the user declines the pipeline and wants the change ad-hoc → S2 flow applies
   (notes updated, still no requirement — Hercules never forces the workflow).

**Expected at the end:** the area's first business-requirements doc + shard with evidence rows
+ anchored tests + updated map/notes; from now on the area behaves like S1/S3.
**Excluded:** no retroactive requirements for the *rest* of the module (only the touched
feature gets one), no full-repo re-scan (light mode deepens only the touched directory), the
note is not deleted (it keeps covering the still-unmanaged remainder of the area).

## Protocol-layout advisory (round 3): split `workflow-protocol.md` into per-phase files?

**Question examined:** replace the single `plugin/protocols/workflow-protocol.md` with
`plugin/protocols/workflow/` containing `workflow-common.md` + `phase-1-discover.md` …
`phase-4-ship.md`, so the orchestrator reads only the relevant phase file.

**Method:** three-advisor debate (pro-split token economist, anti-split simplicity skeptic,
neutral mechanics verifier), each independently measuring against the actual branch files.

### Verdict: DO NOT split (2:1; the supporter's case reduces to headroom, achievable cheaper)

**Facts all three advisors converged on:**

1. **The proposal's mechanical premise is half-right.** `#anchor` references are semantic
   hints, not loaders — Claude's Read returns whole files, so files (not anchors) are the only
   real read-granularity boundary. Splitting *would* bound reads. But—
2. **The bounded quantity is negligible.** Only the orchestrator reads the protocol, roughly
   once per phase (5 reference sites total; sub-agents never read it — a test forbids it).
   Section math: a `common.md` must carry preamble + packet template + role expectations +
   cross-phase guardrails (G6 spans build+ship, G7 spans all) ≈ **~470–520 of the ~950
   tokens — half the file**. Net saving after extra-Read overhead: ~300–500 tokens once per
   phase for Discover/Design/Ship, **~0–150 for Build** — the longest, most spawn-heavy phase
   needs ~90% of the file (G1–G5 + 3 of 4 role rows are Build's). Two of the five proposed
   files would be 45–60-token stubs. `/hercules:workflow` full runs save nothing and pay ~3
   extra Read overheads. Well under 1% of orchestrator context in every case.
3. **The dominant protocol token flow is untouched by the split**: per-spawn packet injection
   (~1.4–1.8k tokens per debate round-set) copies registry rows into every spawn regardless of
   how the source file is laid out.
4. **The split adds a new silent failure class.** Today the packet template, registry rows,
   and role expectations arrive atomically in one Read — a packet can't be composed with the
   template present but rows missing. After a split, "read phase file, skip common" yields a
   malformed packet; "read common, skip phase file" makes the model paraphrase guardrails from
   memory — precisely the verbatim-copy failure the registry exists to prevent. A mid-build
   spec-scoped ship would need **three** files (common + phase-3 + phase-4) where today it
   needs one.
5. **Migration is heavy and contains two silent-weakening traps** (found by two advisors
   independently): `tests/agents/test_agents.py`'s forbidden-pattern `workflow-protocol` no
   longer matches renamed files (the "protocol reaches agents only via the packet" guard dies
   silently), and `tests/budgets/test_instruction_budget.py` uses a **non-recursive**
   `glob("*.md")` on `plugin/protocols/` — a nested `workflow/` dir would silently drop the
   protocol from the orchestrator instruction budget. Plus: `test_workflow_protocol.py` is
   effectively a rewrite (one anchor-resolution test would start passing vacuously), 7
   reference sites to migrate, and thresholds.json needs five invented per-file limits while
   losing the one invariant that matters (total tokens loaded per phase).
6. **The real pressure is genuine but has a one-line fix.** The file sits at 999/1000 against
   a hard CI gate, and this plan's traceability wording (G4, G6, packet Context line) would
   breach it. Resolution: **raise the limit to 1100/warn 1050** (siblings already sit at 1050
   and 1100) and trim preamble/role-expectations prose if still binding. The budget number is
   a self-imposed discipline line, not a physical limit; the discipline survives at 1100.

**Decision applied to this plan:** keep `workflow-protocol.md` as a single file; bump its
thresholds row to 1100/1050 as part of the traceability change; no directory restructure.

**Revisit trigger (recorded for the future):** split only if the protocol legitimately needs
to grow past the point where Build itself would load materially less than the whole file
(~1500+ tokens of genuinely phase-partitioned content). If that day comes, use the debate's
"Design A": single-homed rows (each guardrail lives in exactly one file; cross-phase rows in
common), zero duplication, and the migration must ship in the same PR with: the `rglob` fix in
the instruction-budget test, an updated forbidden-pattern in the agents test, cross-file G-ID
uniqueness assertions, and per-file thresholds with `common.md` gated tightest (it feeds
per-spawn injection).

**Incidental hardening worth doing regardless** (surfaced by the debate): the non-recursive
`glob("*.md")` in `test_instruction_budget.py` is a latent trap for *any* future subdirectory
under `plugin/protocols/` — switch to `rglob` defensively as a tiny part of the traceability
change.

## Open decisions for round 2 (my recommendation first)

1. **Phasing:** greenfield-first (recommended — proves ID/shard foundations on a small
   surface) vs brownfield-first (99% of real repos are code-only — you may want
   `/hercules:trace` in hand immediately) vs both in one release.

   everywhere vs docs-repo-only (code repos untouched, but a service-repo clone carries no
   index).

4. **Scanner deps:** stdlib regex + optional auto-detected ctags/tree-sitter (recommended) vs

   external binary, never bundled.

   visible in raw diffs. Acceptable?
6. **Write-surface expansion** — Hercules would now write committed files into *every* service

7. **Multi-project docs repos** — largely resolved by dated-slug IDs (a collision now needs
